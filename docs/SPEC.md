# khunnect 전체 스펙

> **이 문서의 목적**
> 개발 착수 전에 "무엇을 만들지"를 확정해 두는 단일 기준점.
> 코드·DB·피그마 3자가 어긋나 있어(각자 다른 시점에 만들어짐) 그대로 두면
> 만들 때마다 판단이 갈린다. 여기서 방향을 정하고, 이후 개발은 이 문서를 따른다.
>
> **표기 규칙**
> - ✅ 구현 완료 — 코드에 존재하고 동작이 검증됨
> - 🚧 부분 구현 — 일부만 있거나 데이터가 안 채워짐
> - ❌ 미구현
> - ❓ **미결정** — 개발 전에 정해야 함
> - 🔍 **피그마 확인 필요** — 추측 금지. 실제 프레임을 봐야 함
>
> 최종 갱신: 2026-07-17

---

## 0. 제품 정의

경희대 국제캠퍼스 학생을 위한 **커리큘럼 설계 플랫폼**.
재학생·졸업생이 자신의 커리큘럼/자기개발 경로를 공유하고, 후배가 이를 참고해
방향을 잡는 **선순환**을 만든다.

**선순환 고리 (이게 제품의 축)**
```
선배가 경로를 기록·공개  →  후배가 탐색·열람  →  커피챗으로 연결  →  대화
        (로드맵)              (선배 디렉토리)      (커피챗)         (채팅)
                                     ↓
                        후배도 졸업 후 선배가 되어 경로를 공개
```
어떤 기능을 넣을지 고민될 때 **"이 고리를 강하게 하는가"**를 기준으로 판단한다.

인증 가드는 각 라우트의 `beforeLoad`에서 단 한 번만 수행한다(중복 금지 — CLAUDE.md 참조).

### 1.2 데이터 모델 (전체)

```
colleges ─┬─ departments ─┬─ tracks
          │               ├─ course_catalog ── checked_courses ── profiles
          │               ├─ curriculum_requirements
          │               └─ user_majors ── profiles
profiles ─┬─ bookmarks (user_id, senior_id)
          ├─ coffee_chats (student_id, senior_id, status) ── messages
          └─ roadmaps (user_id UNIQUE, is_public) ── roadmap_items
```

**핵심 제약**
- `user_majors.type` = `major` | `double_major` | `minor` — **한 명이 여러 행**을 가진다.
  조인 순서가 보장되지 않으므로 반드시 `lib/majors.ts`의 `mapMajors`로 정렬해 쓴다.
- `roadmaps.user_id`는 UNIQUE — **유저당 로드맵 1개**.
- `roadmap_items.status` = `done` | `planned` — 회고록·계획표 겸용.
- `coffee_chats.status` = `pending` | `accepted` | `declined` | `cancelled`,
  `(student_id, senior_id)` pending 중복 방지, 자기 자신 신청 차단(CHECK).
- `messages`는 `coffee_chats.status = 'accepted'`일 때만 전송 가능(RLS).

**보안 (마이그레이션 0004)**
`profiles.email`은 anon/authenticated에게 **차단**. 테이블 SELECT를 회수하고
안전한 컬럼만 GRANT한다. ⚠️ 컬럼 단위 REVOKE는 테이블 SELECT가 있으면 무효 —
새 컬럼 추가 시 GRANT 목록에 넣어야 보인다.

## 2. 기능별 스펙

### 2.1 선배 탐색 `/explore` — 🚧 확장 필요

**현재 (✅)**
- `role='alumni'` 프로필 목록, 학과 칩 필터 / 상담가능 토글 / 이름 검색
- 카드: 아바타 · 이름 · 학과(복수전공 포함) · 졸업연도 · 상담가능 뱃지 · 전문분야 · 북마크

**피그마와의 차이 (확장 대상)**
| 항목 | 현재 | 피그마 |
|---|---|---|
| 레이아웃 | 칩 필터 + 균일 그리드 | **학과별 섹션 헤딩** 아래 카드 배치 |
| 분야 필터 | 없음 | **`▾ 모든 분야` 드롭다운** (= 트랙) |
| 상세 진입 | 카드 전체 클릭 | **`프로필 보기` 버튼** |

**❓ 결정 필요**
1. **학과 섹션 vs 칩 필터** — 학과가 늘면 섹션 방식은 페이지가 길어진다.
   현재 시드는 학과 1개(소프트웨어융합학과)뿐이라 섹션이 무의미.
   → **제안: 학과가 5개 이상 될 때까지 칩 필터 유지.** 섹션은 데이터가 쌓인 뒤.
2. **`모든 분야` = 트랙이 맞나?** 🔍 피그마 확인 필요.
   트랙(게임콘텐츠/데이터사이언스/로봇·비전/융합리더)이라면 §6.2 선행 필요.
3. 복수전공자를 학과 섹션 방식에서 **양쪽에 중복 노출**할지 주전공에만 놓을지.
   → **제안: 양쪽 노출.** 후배는 "그 학과 선배"를 찾는 것이므로.

### 2.3 커피챗 — ✅ 동작, 🔍 디자인 미확인

**현재**
- 학생이 선배에게 신청(메시지 포함) → 선배가 수락/거절, 학생이 취소
- 마이페이지에서 받은/보낸 신청 관리
- pending 중복 차단, 자기 신청 차단

**🔍 피그마 확인 필요** — 피그마에 커피챗 전용 프레임이 안 보였다.
nav의 `메시지함`이 커피챗인지 채팅인지 불명확.

**❓ 결정 필요**
1. **일정 조율이 필요한가?** 피그마에 `mypage_calendar_01~04`가 있는 걸 보면
   커피챗에 **날짜/시간** 개념이 있을 가능성이 크다. 현재 `coffee_chats`에는
   시간 컬럼이 없다(`created_at`, `responded_at`만).
   → 캘린더를 만들려면 **스키마 변경 필요**(§2.6).
2. 수락 후 알림이 없다 — 선배가 수락해도 학생은 새로고침해야 안다(§2.7).

### 2.4 실시간 채팅 — ✅ 동작, 🔍 디자인 미확인

**현재**
- `messages` + Supabase Realtime 구독
- 수락된 커피챗에서만 "채팅 열기" → 모달
- RLS: 당사자만 조회·전송, 미수락 채팅엔 전송 불가

**❓ 결정 필요**
- 모달이 맞나, 전용 페이지(`/messages`)가 맞나. nav에 **`메시지함`** 버튼이
  이미 있는 걸 보면 전용 페이지 의도일 수 있다. 🔍 확인 필요.
- 읽음 표시 / 안 읽은 개수 — 현재 없음. 필요하면 `messages.read_at` 추가.

### 2.5 커리어 로드맵 `/roadmap` — ✅ 구현, ⚠️ 설계 근거 약함

> ⚠️ **주의**: 이 기능은 피그마 설계 없이 구현됐다(프레임이 `road…` 몇 개뿐).
> 데이터 모델과 UI는 **개발 중 내린 판단**이지 디자인 확정본이 아니다.
> 피그마에 로드맵 프레임이 따로 있다면 **먼저 대조할 것.** 🔍

**현재 모델**
```
roadmaps(user_id UNIQUE, title, summary, is_public=false)
  └ roadmap_items(year 1~6, semester 1|2, type, title, description, status)
     type: 수강 | 인턴 | 프로젝트 | 대외활동 | 자격증 | 취업 | 기타
     status: done(이미 함) | planned(할 예정)
```
- 기본 **비공개** — 본인이 켜야 후배에게 보임
- 항목 RLS는 부모 로드맵의 공개 여부를 따라감 (검증됨)

**❓ 결정 필요**
1. **발견 경로가 없다.** 지금은 선배 상세로 들어가야만 로드맵을 본다.
   피그마 nav에 `커리어 로드맵 ▾` 드롭다운이 있는 걸 보면 **탐색 화면**이
   의도됐을 수 있다. → **제안: `/roadmap/explore` 신설** (공개 로드맵 목록·필터).
   이게 선순환에 가장 크게 기여한다.
2. **유저당 1개**가 맞나? "게임 개발자 경로" / "대학원 경로"처럼 여러 개를
   그리고 싶을 수 있다. → **제안: 1개 유지.** 자기 경로는 하나뿐. 복잡도만 늘어난다.
3. 커리큘럼(`checked_courses`)과 연동할지 — 체크한 과목을 로드맵 `수강` 항목으로
   자동 채우기. → **제안: 나중.** 수동 입력이 부담되면 그때.

### 2.6 마이페이지 캘린더 — ❌ 미구현, 피그마 4화면

**현재**: `MyPage.tsx:264` — "캘린더 기능은 준비 중입니다" 스텁.

**🔍 피그마 확인 필수** — `mypage_calendar_01~04`. 4화면이나 설계된 걸 보면
가벼운 기능이 아니다. **무엇의 캘린더인지**부터 확인해야 한다.

**❓ 결정 필요 (가장 큰 미결정)**
캘린더의 정체가 셋 중 무엇인가:
- **(a) 커피챗 일정** — 선배와 만날 날짜. → `coffee_chats`에 `scheduled_at` 추가 필요.
- **(b) 학사 일정** — 수강신청·시험 기간 등. → 새 테이블 + 데이터 출처 필요(크롤링?).
- **(c) 개인 할 일** — 로드맵 `planned` 항목의 마감일. → `roadmap_items`에 `due_date` 추가.

→ **셋 다 스키마 변경이 필요하고 방향이 완전히 다르다. 확인 전 착수 금지.**
→ **제안: (a)가 가장 유력**하고 선순환에도 기여한다. (b)는 외부 데이터 의존이라 위험.
