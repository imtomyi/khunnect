## 1. OrbStack — 리눅스 개발 환경

실제 운영 서버는 대부분 Linux를 기반으로 한다. Docker 이미지, Kubernetes 노드, CI 서버 모두 Linux 위에서 동작하므로, macOS에서 개발할 경우 "로컬에서는 되는데 서버에서는 안 되는" 간극이 발생한다. 이 간극을 줄이기 위해 개발 환경 자체를 Linux로 옮겼다. 초기에 **UTM**(QEMU 기반 가상화)을 검토하였으나, macOS와 VM 사이의 파일 공유·SSH·포트포워딩을 매번 설정해야 하고 재시작 시 IP가 바뀌어 환경 전환 비용이 컸다. 대안으로 도입한 **OrbStack**은 macOS 전용 경량 Linux Machine으로, 이 문제들을 대부분 해소하였다.

```bash
brew install orbstack
orb            # Linux Machine 진입
orb list       # ubuntu / running / arm64 / 8.3GB

# Linux 머신 안에서 개발 도구 설치
sudo apt-get update && sudo apt-get upgrade -y
nvm install 20            # Node.js 20
sudo apt-get install -y git
```

UTM과 비교하면 차이가 분명하였다.

| 항목 | UTM | OrbStack Linux Machine |
|------|-----|------------------------|
| 파일 공유 | 수동 VirtFS 설정 | `/mac/...` 경로 자동 마운트 |
| 포트포워딩 | 수동 설정 | macOS localhost 자동 연결 |
| Docker | VM 내 별도 설치 필요 | 소켓 공유로 즉시 사용 |
| VS Code SSH | IP 직접 지정(재시작 시 변동) | 호스트 `orb` 고정 |
| 성능 기반 | QEMU 에뮬레이션 | Virtualization.framework(네이티브급) |

VS Code Remote SSH의 호스트를 `orb`로 고정하여 macOS에서 편집하고 Linux에서 실행하는 흐름을 구성하였다. 핵심 주의점은 **OrbStack Linux Machine이 ARM64(aarch64) 아키텍처**라는 것이다. 이후 모든 도구 설치 시 반드시 arm64 바이너리를 사용해야 하며, amd64 바이너리는 `Dynamic loader not found: /lib64/ld-linux-x86-64.so.2` 오류로 실행되지 않는다. 이 경험은 4번 Kubernetes 설치에서 실제 문제로 다시 나타났다.

## 2. Vite — 프론트엔드 빌드 도구

React 19 + TypeScript 프론트엔드의 빌드 도구로 **Vite 8**을 사용하였다. Vite는 두 가지 역할을 한다. 개발 시에는 HMR(Hot Module Replacement) 개발 서버로서 파일 저장 시 변경된 모듈만 즉시 교체하여 브라우저에 반영하고, 프로덕션에서는 번들러로서 TypeScript 컴파일·모듈 묶음·최적화를 거쳐 `/dist` 정적 파일을 생성한다. Vite 8은 기존 esbuild/Rollup 조합에서 Rust 기반 **Rolldown** 번들러로 전환되어 빌드가 빠르며, OrbStack Linux Machine에서 개발 서버 콜드 스타트는 약 541ms로 측정되었다.

```bash
npm run dev     # 개발 서버 (HMR) — http://localhost:5173
npm run build   # tsc -b && vite build → dist/ 생성
```

`npm run build`는 타입 체크(`tsc`)를 먼저 수행한 뒤 번들링하므로, **타입 오류가 있으면 빌드가 중단**되어 배포 이전 단계에서 타입 안정성을 강제하는 안전장치가 된다.

빌드 최적화로 `manualChunks`를 적용하여, 자주 바뀌지 않는 라이브러리를 애플리케이션 코드와 분리된 별도 청크로 묶었다. Rolldown은 객체 형태를 허용하지 않으므로 반드시 함수 형태로 작성해야 한다.

```ts
// vite.config.ts
manualChunks: (id: string) => {
  if (id.includes('node_modules/react'))      return 'vendor-react'
  if (id.includes('@tanstack/react-router'))  return 'vendor-router'
  if (id.includes('@tanstack/react-query'))   return 'vendor-query'
  if (id.includes('@supabase'))               return 'vendor-supabase'
}
```

또한 모든 페이지 컴포넌트를 `React.lazy()`로 동적 import하여 코드 스플리팅을 적용하였다. 사용자가 해당 라우트에 진입할 때 비로소 해당 청크를 내려받으므로, 초기 로딩에 불필요한 코드가 포함되지 않는다. 빌드 결과는 **234개 모듈 / 19개 청크 / 최대 청크 195.9KB / 타입 오류 0**이었다. 벤더 청크는 한 번 캐시되면 앱 코드를 수정해도 다시 받지 않으므로, 재배포 시 사용자가 내려받는 용량이 최소화된다.

## 3. Docker — 컨테이너화

Vite가 생성한 정적 결과물을 어디서나 동일하게 실행되도록 **Docker** 이미지로 패키징하였다. 이때 빌드 환경(Node)과 실행 환경(nginx)을 분리하는 **멀티스테이지 빌드**를 적용하였다. 단일 스테이지로 구성하면 Node 런타임과 node_modules가 최종 이미지에 모두 포함되어 크기가 불필요하게 커지기 때문이다.

```dockerfile
FROM node:20-alpine AS builder      # 1단계: 빌드
WORKDIR /app
COPY package*.json ./               # 의존성 파일만 먼저 복사 → 레이어 캐시
RUN npm ci
COPY . .
RUN npm run build                   # → /app/dist

FROM nginx:alpine                   # 2단계: 서빙
COPY --from=builder /app/dist /usr/share/nginx/html
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
```

핵심 효과는 세 가지이다.

- **이미지 경량화** — 최종 이미지에서 Node 런타임·node_modules를 제외하고 nginx와 정적 파일만 포함하여, 600MB 이상이 되었을 이미지를 **94.1MB**로 축소하였다.
- **레이어 캐시 전략** — Dockerfile의 각 명령은 하나의 레이어이며, 변경된 레이어부터 이하 전부가 재실행된다. `package.json`을 소스보다 먼저 복사함으로써, 의존성이 바뀌지 않으면 `npm ci`(가장 느린 단계)를 재실행하지 않는다. 변경 빈도가 낮은 명령을 상단에 배치하는 것이 빌드 속도 최적화의 핵심이다.
- **서빙 계층 일관성** — 2단계의 `nginx:alpine`이 정적 파일을 HTTP로 서빙한다. 동일한 이미지가 로컬 Docker·Kubernetes·시놀로지 전 배포 단계에서 환경과 무관하게 동일하게 동작하였다.

```bash
docker build -t khunnect-frontend:latest ./frontend
docker run -p 3000:80 khunnect-frontend:latest
curl -I http://localhost:3000      # HTTP/1.1 200 OK / Server: nginx
```

이미지는 클래스, 컨테이너는 그 인스턴스에 비유할 수 있으며, 컨테이너 종료 시 내부 변경 사항은 사라진다.

## 4. Kubernetes (minikube) — 컨테이너 오케스트레이션

단일 컨테이너 실행만으로는 종료 시 수동 재시작이 필요하고 다수 서버 분산이 어렵다. 이를 넘어 원하는 상태를 선언적으로 정의하고 컨테이너를 자동 복구하기 위해 **minikube**로 로컬 Kubernetes를 경험하였다. 핵심 개념은 다음과 같다.

- **Pod** — 컨테이너를 감싸는 최소 배포 단위
- **Deployment** — 지정한 replica 수를 유지하며 Pod 종료 시 자동 재생성
- **Service** — Pod IP 변동과 무관한 고정 네트워크 엔드포인트 (NodePort 30080)

설치 단계에서 아키텍처 문제를 겪었다. OrbStack은 ARM64이므로 amd64 바이너리는 `Dynamic loader not found` 오류가 발생하였고, arm64 바이너리로 재설치하여 해결하였다.

```bash
curl -LO ".../bin/linux/arm64/kubectl"   # 반드시 arm64
minikube start --driver=docker
```

대표 트러블슈팅은 **ImagePullBackOff**였다. minikube는 호스트와 별개의 Docker daemon을 사용하므로, 호스트에 빌드한 이미지를 찾지 못해 발생한다. `eval`로 현재 쉘을 minikube의 daemon으로 전환한 뒤 빌드해야 한다.

```bash
eval $(minikube docker-env)        # 쉘을 minikube daemon으로 전환
docker build -t khunnect-frontend:latest ./frontend
kubectl apply -f k8s/frontend-deployment.yaml
kubectl get pods                   # frontend-... 1/1 Running ✅
minikube service frontend-service --url   # http://192.168.49.2:30080
```

`eval $(minikube docker-env)`는 `export DOCKER_HOST=...` 형태의 출력을 현재 쉘에 즉시 적용하여, 이후 `docker` 명령이 minikube 내부 daemon을 가리키게 하는 패턴이다. 이를 통해 빌드한 이미지를 별도 레지스트리 없이 클러스터에서 바로 사용할 수 있었으며, nginx가 Kubernetes 위에서 React 앱을 정상 서빙함을 확인하였다.

## 5. Supabase — 서버리스 백엔드

별도의 Express/Node.js API 서버를 구축하지 않고 인증·DB·접근제어를 통합 제공하는 BaaS(Backend as a Service)로 **Supabase**를 사용하였다. Supabase는 PostgreSQL·Auth·Storage·RLS를 하나로 제공하므로, React에서 Supabase JS SDK를 통해 데이터베이스에 직접 접근할 수 있다.

```ts
// src/lib/supabase.ts
export const supabase = createClient(
  import.meta.env.VITE_SUPABASE_URL,
  import.meta.env.VITE_SUPABASE_ANON_KEY)
```

핵심은 **RLS(Row Level Security)** 이다. 행 단위 접근 권한을 PostgreSQL이 강제하므로, 프론트엔드가 DB에 직접 접근하더라도 본인 데이터만 읽고 쓸 수 있다.

```sql
CREATE POLICY "own profile" ON profiles
  FOR SELECT USING (auth.uid() = id);
CREATE POLICY "own checked courses" ON checked_courses
  FOR ALL USING (auth.uid() = user_id);
```

`auth.uid()`는 현재 로그인한 사용자의 UUID를 반환하는 Supabase 내장 함수로, 이를 활용하면 애플리케이션 코드에 별도의 권한 체크 로직을 작성하지 않아도 된다. 한편 `service_role` 키는 RLS를 우회하는 관리자 키이므로 **프론트엔드 코드에 절대 포함해서는 안 된다.**

사용자 경험을 위해 **낙관적 업데이트(Optimistic Update)** 패턴을 적용하였다. 커리큘럼 과목 체크 시 DB 응답을 기다리면 UI가 멈춘 것처럼 보이므로, 캐시를 먼저 즉시 갱신하고 DB는 백그라운드에서 동기화한다.

```ts
queryClient.setQueryData(['checked_courses', userId], (prev = []) =>
  isChecked ? prev.filter(x => x !== id) : [...prev, id])
// 이후 supabase.from('checked_courses').insert/delete 로 백그라운드 동기화
```

이로써 사용자 입장에서 응답이 즉각적으로 느껴지면서도 데이터는 서버와 일관되게 유지된다.

## 6. Synology DSM — 자가 서버 배포

보유 중인 **Synology NAS**를 실제 배포 서버로 운영하였다. DSM(DiskStation Manager)은 Linux 커널 기반(커널 3.10.108, x86_64) 운영체제이며, SSH로 접속하면 Linux 환경을 그대로 활용할 수 있다. 하드웨어는 RAM 7.7GB·디스크 여유 2.5TB로, 정적 파일을 nginx로 서빙하는 데 충분한 사양임을 확인하였다.

서버를 네트워크에 노출하기에 앞서 SSH 보안을 다단계로 강화하였다. 보안은 단일 설정이 아니라 여러 계층의 누적임을 체감하였다.

1. **포트 변경** — 기본 22번은 자동화 봇의 상시 스캔 대상이므로 비표준 포트로 변경
2. **키 인증** — ed25519 키 쌍을 생성·등록하여 무차별 대입(brute force) 공격을 방어
3. **비밀번호 비활성화** — `PasswordAuthentication no`로 비밀번호 로그인을 완전 차단
4. **방화벽·자동 차단** — SSH 도달 범위를 내부망으로 제한하고 반복 실패 IP를 자동 차단

```bash
ssh-copy-id -i ~/.ssh/id_ed25519.pub -p [포트] [계정]@[내부IP]
chmod 700 ~/.ssh && chmod 600 ~/.ssh/authorized_keys   # 권한 느슨하면 키 거부
sudo sed -i 's/^#*PasswordAuthentication.*/PasswordAuthentication no/' \
  /etc/ssh/sshd_config && sudo synosystemctl restart sshd
```

배포는 git 미설치 환경이라 `tar`를 SSH 통로로 흘려보내 코드를 전송한 뒤, 시놀로지가 x86_64 네이티브이므로 직접 빌드·실행하였다.

```bash
tar czf - --exclude node_modules frontend | ssh -p [포트] [계정]@[내부IP] \
  'mkdir -p app && tar xzf - -C app --strip-components=1'
sudo docker build -t khunnect-frontend:latest .
sudo docker run -d -p 8080:80 --restart unless-stopped khunnect-frontend:latest
curl -I http://localhost:8080      # HTTP/1.1 200 OK
```

이로써 첫 자가 서버 배포를 완료하였다. 가상 환경이 리눅스를 "배우는 공간"이었다면, 시놀로지는 보안·아키텍처·환경 차이를 직접 다루며 리눅스로 무언가를 "운영하는 공간"이었다.

## 7. GitHub Actions — CI/CD

코드 push 시 자동으로 빌드 가능 여부를 검증하는 **CI(지속적 통합)** 를 구성하였다. main 브랜치에 push하거나 PR을 생성하면 워크플로우가 실행된다.

```yaml
# .github/workflows/ci.yml
on:
  push: { branches: [main] }
  pull_request: { branches: [main] }
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20, cache: 'npm' }
      - run: cd frontend && npm ci          # 의존성 재현 설치
      - run: cd frontend && npm run build    # 타입 체크 + 번들 (오류 시 중단)
      - run: docker build -t khunnect-frontend ./frontend   # 이미지 빌드 검증
```

세 단계(`npm ci → build → docker build`)가 순차 실행되어, 타입 오류나 빌드 실패가 있는 코드가 main에 병합되는 것을 사전에 차단한다.

**자동 배포(CD)** 는 계획 단계이다. 가정용 서버는 공유기 뒤에 위치하여 외부(클라우드)에서 직접 SSH 접근이 불가능하다. 따라서 포트를 외부에 개방하는 대신, 시놀로지가 능동적으로 바깥에 접속하여 변경 사항을 가져오는 **outbound 방식**으로 설계한다.

```
git push → GitHub Actions가 amd64 이미지 빌드 → ghcr.io(레지스트리)에 push
        → 시놀로지의 Watchtower가 이를 감지·pull → 컨테이너 자동 교체 → 라이브
```

이 구조는 인바운드 포트를 추가로 개방하지 않으므로 6번에서 강화한 SSH 보안을 그대로 유지한다. 빌드는 클라우드에서 수행되어 시놀로지 부하가 없고, 레지스트리에 이미지 이력이 축적되어 롤백이 용이하다. 결과적으로 `git push` 한 번으로 빌드 검증부터 배포까지 무인으로 이어지는 파이프라인이 완성된다. 이처럼 7개 소프트웨어는 개발(OrbStack·Vite)부터 패키징(Docker)·오케스트레이션(Kubernetes)·백엔드(Supabase)·자가 배포(Synology)·자동화(GitHub Actions)까지 하나의 흐름으로 연결되었다.
